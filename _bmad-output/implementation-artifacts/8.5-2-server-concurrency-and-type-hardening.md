# Story 8.5.2: Server-Side Concurrency & Type Hardening

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As an engineer,
I want session-manager broadcast plumbing, per-user state lifecycles, and wire-format types hardened before Phase 2 introduces wallet/XP/honor at match end,
so that the new Epic 9 surfaces inherit correct primitives instead of inheriting the same risk shapes.

## Acceptance Criteria

1. **AC1 — `TeamStringForIndex` safe on out-of-range (resolves D113)**

   **Given** `game.TeamStringForIndex` (Go) and `teamStringForIndex` (TS) receive a team index,
   **When** the index is outside `{0, 1}`,
   **Then** the Go variant returns `""` (empty string sentinel), **never panics**,
   **And** the TS variant is overloaded — callers with `0 | 1` typed arguments still get `TeamString` (no null change); callers with raw `number` get `TeamString | null`,
   **And** Zod schemas for `winnerTeam` fields in `MatchEndPayloadSchema`, `TrickResolvedPayloadSchema`, and `DeclarationsResolvedPayloadSchema.winnerTeam` tighten to `z.union([z.literal(0), z.literal(1)])` (or `.nullable()` where the field is nullable) so the contract test rejects garbage at parse time.

2. **AC2 — `OutcomeReason` typed as named Go type + TS union (resolves D116)**

   **Given** the WS `OutcomeReason` field is emitted by the server in `event:match_end`,
   **When** the client parses the event,
   **Then** the Go field `MatchEndPayload.OutcomeReason` is typed as `OutcomeReason` (named `string` type with constants), not raw `string`,
   **And** the constants defined are `OutcomeReasonSurrender = "surrender"`, `OutcomeReasonTimeout = "timeout"`, `OutcomeReasonAbandonment = "abandonment"` — Epic 9 will append `"insolvency"` and `"honor_eject"`; the type is designed to accept future appends,
   **And** the TS `outcomeReason` field on `MatchEndPayload` changes from `"surrender"` to `"surrender" | "timeout" | "abandonment" | "natural"`,
   **And** `MatchEndPayloadSchema.outcomeReason` changes from `z.literal("surrender").optional()` to a `z.union([...])` covering all four values,
   **And** both `server/internal/ws/events.go` and `client/src/shared/types/wsEvents.ts` are updated in **the same commit** (project rule — WS contract sync).

3. **AC3 — `lastEmoteAt` map bounded; cross-match bleed eliminated (resolves D105 + D106)**

   **Given** a user is removed from `session.Manager` (match end, leave, disconnect window expiry),
   **When** `RemoveSession` runs,
   **Then** the per-user `lastEmoteAt` entry in `emote.Handler` is cleared via a `UserRemovedHook` callback,
   **And** future cross-match emotes from that user respect a fresh rate-limit window in the next match,
   **And** the per-process `lastEmoteAt` map is bounded by current-active users (never lifetime users),
   **And** the new `AddUserRemovedHook(fn func(userID uint))` method on `session.Manager` is reusable for Epic 9's per-user state (wallet rate-limit, daily-claim cooldown) — hooks are called **after** the manager write-lock is released, preventing deadlocks.

4. **AC4 — Stale participants broadcast (D109 formally accepted)**

   **Given** `session.Manager.MatchParticipantsByUser` returns a participant slice under RLock,
   **When** `RemoveSession` (write lock) interleaves between the slice capture and the `BroadcastToUsers` call in the emote handler,
   **Then** this race is **formally accepted** — `BroadcastToUsers` silently skips disconnected clients; the worst-case consequence is a tail-of-match emote reaching users who just returned to the lobby (benign),
   **And** a tracking comment is added at the broadcast call site in `emote/handler.go` explaining the acceptance decision per deferred-work.md D109,
   **And** `deferred-work.md` is annotated `→ Formally accepted in Story 8.5-2`.

5. **AC5 — `ParseUint` → `uint` truncation fixed (resolves D86)**

   **Given** the auth `:id` route parameter is parsed via `strconv.ParseUint(c.Param("id"), 10, 64)` in `user/handler.go`,
   **When** the result is compared against `authUserID` (a `uint` from the Echo context),
   **Then** the cast is replaced in the three user-handler functions: `uint(paramID) != authUserID` → `paramID != uint64(authUserID)` — this makes the comparison 64-bit-safe on any platform,
   **And** a compile-time platform assertion is added so a 32-bit build fails before the handler can reach runtime (belt-and-suspenders; the fixed comparison already handles the correctness issue).

    // AddUserRemovedHook registers fn to be called (outside the manager lock)
    // for each playerID when RemoveSession tears down a session.
    // Reusable for Epic 9 per-user state (wallet rate-limit, daily-claim cooldown).
    func (m *Manager) AddUserRemovedHook(fn func(userID uint)) {
        m.mu.Lock()
        defer m.mu.Unlock()
        m.userRemovedHooks = append(m.userRemovedHooks, fn)
    }

    Edit `RemoveSession` to capture the removed player IDs and snapshot the hooks slice, then call hooks **after** unlocking:
    ```go
    func (m *Manager) RemoveSession(roomID uint) {
        m.mu.Lock()
        var removedIDs []uint
        var hooks []func(userID uint)
        if session, ok := m.sessions[roomID]; ok {
            session.mu.Lock()
            session.closed = true
            session.cancelTurnTimer()
            session.cancelReconnectTimer()
            session.mu.Unlock()
            for _, uid := range session.playerIDs {
                delete(m.userToRoom, uid)
                removedIDs = append(removedIDs, uid)
            }
            delete(m.sessions, roomID)
            slog.Info("session: removed", "roomID", roomID)
        }
        hooks = append(hooks, m.userRemovedHooks...) // snapshot under lock
        m.mu.Unlock()

        // Call hooks after releasing the manager lock to prevent deadlocks.
        for _, uid := range removedIDs {
            for _, fn := range hooks {
                fn(uid)
            }
        }
    }
    ```

## Dev Notes

### What This Story IS — and What It ISN'T

This is a **type-hardening and lifecycle-hygiene** story, not a feature story. Four concrete correctness fixes and one formal acceptance:

- **D113**: Prevent a panic in a hot-path function used on every scored hand.
- **D116**: Give `OutcomeReason` the same type safety in Go that Epic 9 requires (settlement discriminator will use this field).
- **D105/D106**: Prevent the `lastEmoteAt` map from growing unbounded and leaking cross-match rate-limit state. The `AddUserRemovedHook` mechanism is reusable — Epic 9 will need the same hook for wallet rate-limits.
- **D109**: Accepted. The emote race is benign; fixing it requires a session-held broadcast interface that is premature at this stage.
- **D86**: Fix a comparison that would incorrectly pass on 32-bit platforms.

**No new HTTP endpoints. No new WS events. No SQL migration. No new domain packages. No new Zustand stores.**

### Critical Guardrails

1. **AC2 WS contract sync**: `events.go` and `wsEvents.ts` MUST be in the same commit. The contract test (`events_contract_test.go`) will catch drift.
2. **AC3 hooks called after unlock**: The `UserRemovedHook` functions are called **after** `m.mu.Unlock()`. Do NOT call them while holding the write lock — the emote handler's `RemoveUser` acquires `h.mu` (its own lock), which is fine, but future hooks may do arbitrary work. The snapshot pattern (`hooks = append(hooks, m.userRemovedHooks...)` under lock, then call after unlock) is correct.
3. **AC1 TS overloads**: The `teamStringForIndex` overload must be declared before the implementation. TypeScript requires the overload signatures appear before the implementation signature. `ScorePanel.tsx:52` uses `showBonus.team` which is typed `0 | 1` — it will resolve to the narrow overload and require no changes.
4. **No scope creep**: D87/D88 (stats filter desync) and D124 (lockfile script) are conditional on this story touching `match/gorm_repo.go` or CI failures. This story does NOT touch `match/gorm_repo.go`, so D87/D88 are **out of scope**. Skip them.

### Architecture Patterns to Follow

- **Named string types for discriminators**: `OutcomeReason` follows the pattern of `ws.EmoteID` and `game.Suit` — `type OutcomeReason string` with `const` block. Future Epic 9 appends (`"insolvency"`, `"honor_eject"`) just add constants without changing the type.
- **Hook registration over injection**: `AddUserRemovedHook` follows the pattern established by `SetRoomUpdater` in the same manager. Hooks are registered after construction in `main.go`. The manager does not know about `emote.Handler` at compile time.
- **Hooks called outside the lock**: Same discipline as goroutine-dispatch patterns in session manager. The manager's write lock is already unlocked before the I/O-heavy path (broadcast); hooks follow the same principle.
- **Zod strictObject**: All schemas use `z.strictObject` — the tightened `winnerTeam` fields maintain this. Adding the union does not change the strict-object guarantee.
- **Test helper reuse**: New tests use existing fake infrastructure (`fakeGame`, `hubSpy` in emote tests; existing user-handler helpers for AC5). No new mock frameworks.
