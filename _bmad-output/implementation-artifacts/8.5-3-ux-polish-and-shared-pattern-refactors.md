## Story

As a player,
I want reveals, overlays, prompts, and team labels to behave consistently across reconnect, phase transition, and OS-level setting changes,
so that the in-match experience is coherent before Phase 2 introduces more reveal/overlay surfaces (settlement, daily-reward, insolvent-eject).

1. **AC1 — Shared `useReducedMotion` hook used by all reveal/animation components** _(resolves: D68 + D122)_

   **Given** the OS-level "Reduce motion" setting toggles mid-session,
   **When** any of the snapshot-once `useMemo([])` call sites — `BelotReveal`, `DeclarationReveal`, `RoomLobby`, `DealAnimation`, `CapotAnimation`, `ScoreReveal`, `ScorePanel`, `ReshuffleAnimation` — renders next,
   **Then** it reads the live value via a shared [client/src/shared/hooks/useReducedMotion.ts](client/src/shared/hooks/useReducedMotion.ts) hook that subscribes to the `prefers-reduced-motion` media-query `change` event,

    ```ts
    import { useEffect, useState } from "react";

    /**
     * Subscribes to the `prefers-reduced-motion: reduce` media query and returns
     * the live value. Reactive — toggling the OS-level setting mid-session
     * updates consumers on the next render. SSR-safe (returns false if window
     * or matchMedia is undefined). Replaces the snapshot-once useMemo([])
     * pattern used in 8+ reveal/animation components (D68 + D122).
     */
    export function useReducedMotion(): boolean {
      const [reduced, setReduced] = useState<boolean>(() => {
        if (typeof window === "undefined" || typeof window.matchMedia !== "function") {
          return false;
        }
        return window.matchMedia("(prefers-reduced-motion: reduce)").matches;
      });

      useEffect(() => {
        if (typeof window === "undefined" || typeof window.matchMedia !== "function") {
          return;
        }
        const mql = window.matchMedia("(prefers-reduced-motion: reduce)");
        const listener = (e: MediaQueryListEvent) => setReduced(e.matches);
        mql.addEventListener("change", listener);
        return () => mql.removeEventListener("change", listener);
      }, []);

      return reduced;
    }
    ```

### Frontend Flow — Key Patterns Touched

#### AC1 — `useReducedMotion` hook subscription pattern

```text
Component mounts
  └─ useReducedMotion()
       └─ useState initializer reads window.matchMedia(...).matches  ← initial value
       └─ useEffect on mount: mql.addEventListener("change", listener)
       └─ Component re-renders on each MediaQueryList change event
  └─ Component unmounts
       └─ useEffect cleanup: mql.removeEventListener("change", listener)
```

OS-level setting toggle → MediaQueryList fires `change` → `setReduced(e.matches)` → component re-renders with new value. The same hook instance behind 8 callers means the project has a single subscription point per component instance.

#### AC2 — Reveal reset on reconnect snapshot

```text
Player disconnects mid-reveal
  └─ declarationReveal stays in store with the pre-disconnect payload
  └─ WS connection closes
  ...
Player reconnects
  └─ Server sends event:game_state snapshot (no reveal payload field)
  └─ useWsDispatch.dispatchGameEvent(EVENT_GAME_STATE)
  └─ store.setDeclarationReveal(null)  ← AC2 fix
  └─ store.setBelotReveal(null)        ← AC2 fix
  └─ store.setGameState(snapshot)
  └─ DeclarationReveal unmounts on next render (declarationReveal === null)
  └─ Reveal timer's useEffect cleanup fires, no orphan setTimeout
```

The reveal payload is purely visual; the server explicitly does not replay the cardId-bearing event on reconnect. AC2 closes the loop by treating the reconnect as a clean visual slate.

#### AC3 — Overlay-active reveal gate

```text
GamePage renders
  └─ derive isOverlayActive = isPaused || matchEndData !== null || matchAbandonedData !== null || phase === "disconnected"
  └─ render PauseOverlay (z-20) if isPaused
  └─ render ReconnectOverlay (z-20) if abandoned/disconnected
  └─ render DeclarationReveal (z-20) ONLY IF declarationReveal && !isOverlayActive  ← AC3 fix
  └─ render BelotReveal (z-20) ONLY IF belotReveal && !isOverlayActive              ← AC3 fix
```

The reveal's internal `setTimeout` continues to run while the overlay is up. When the overlay dismisses, if the timer has already fired, the reveal `null`s out via `onComplete` and never appears. If it hasn't, the next render path remounts it (provided the reveal payload is still in the store — which it is, since `onComplete` is the only thing that sets it to `null`).

#### AC4 — Viewer-team-first column ordering

```text
MatchResult render
  └─ viewerTeam derived from myPlayerSeat in GamePage, passed in
  └─ if viewerTeam === "teamA": render <ScoreColumn team="teamA" /> first, then teamB
  └─ if viewerTeam === "teamB": render <ScoreColumn team="teamB" /> first, then teamA
  └─ Each ScoreColumn carries its underlying team's data-testid + data-team
  └─ Visible label is Us/Them (already implemented; preserved)
```

`MatchHistory` and `ReconnectOverlay` already follow this convention. `MatchResult` was the lone outlier per D114.

#### AC8 — Emote picker store-backed cooldown

```text
Today (broken):
  EmotePickerButton mounts (phase=playing)
    └─ useState lastSentAt = 0
    └─ user clicks emote → setLastSentAt(Date.now()) → cooldown 3s
  Phase transitions playing → match_end → dealing
    └─ Picker unmounts, lastSentAt: 5_000_000_000 lost
    └─ Picker remounts, lastSentAt: 0
    └─ User clicks emote within 3s of original → no client cooldown shown
    └─ Server rejects via authoritative rate-limit → silent UX disconnect

After AC8:
  EmotePickerButton mounts (phase=playing)
    └─ const lastSentAt = useGameStore(s => s.lastEmoteSentAt)

    └─ Cooldown is fresh per match
```

`performance.now()` provides a monotonic clock anchored to page-load time. NTP step / OS sleep cannot move it backward.

### Edge Cases & Anti-Patterns to Avoid

- **DO NOT** introduce a feature flag for any of these fixes. They are unconditional UX correctness improvements.
- **DO NOT** rewrite reveal components to be phase-aware (AC3). The components stay presentational; the gate lives in `GamePage.tsx`. Phase awareness in a reveal component leads to prop drilling and tighter coupling for no benefit.
- **DO NOT** change the public signature of `useFocusTrap` for non-Escape callers (AC7). The new `onEscape` is opt-in; every existing call site must remain identical.
- **DO NOT** clear `lastEmoteSentAt` on `event:game_state` (AC8). Cross-match isolation is enforced by `clearGame()` on lobby return; per-match isolation is the server's authoritative rate-limit. The store-backed value only needs to survive in-match remounts.
- **DO NOT** persist `lastEmoteSentAt` to localStorage. It is in-session ephemeral state; surviving a full reload is undesirable (the user may have been away for hours).
- **DO NOT** add reveal-pause/queue logic (AC3). The render gate is sufficient; queue logic is a Phase 2 reveal-queue story.
- **DO NOT** remove the existing `motion-safe:` Tailwind classes when refactoring to `useReducedMotion` (AC1). The `motion-safe:` variant is the CSS-level guard; the hook is the JS-level guard. Both layers stay.
- **DO NOT** modify English (`en.json`) i18n strings (AC9). The native-speaker review is Serbian-only.
- **DO NOT** rename any identifier in Go test code (AC11 D126). Prose / comment cleanup only. The lockfile script is the safety net.
- **DO NOT** touch `useReconnectionRedirect` or `axiosClient` (AC2). The reset is at the dispatch layer where the snapshot arrives.
- **DO NOT** extend AC7 to `TrumpPrompt`. Trump bidding has no "decline" — the existing `Pass` button is the correct semantic. Adding Escape→Pass would conflate two actions.
