   **Then** the staleness guard in [server/internal/session/manager.go:861-868](server/internal/session/manager.go#L861-L868) MUST trip on `session.timerGeneration != generation`,
   **And** the cancel callsite is responsible for bumping `timerGeneration` so a future patch that adds work above the `handleTimerExpiry` switch cannot run after a cancel.

   **Required edit pattern:** move generation-bump duty into [server/internal/session/timer.go](server/internal/session/timer.go) `cancelTurnTimer()` so every cancel ‚Äî not only the ones that immediately re-arm ‚Äî increments the counter. Remove the now-redundant `session.timerGeneration++` lines that immediately follow `cancelTurnTimer()` in `manager.go` and `reconnect.go` (do not let the counter double-increment per cancel/restart pair). Audit existing callsites: [manager.go:161](server/internal/session/manager.go#L161), [manager.go:208-209](server/internal/session/manager.go#L208-L209), [manager.go:295](server/internal/session/manager.go#L295), [manager.go:847-848](server/internal/session/manager.go#L847-L848), [reconnect.go:102-106](server/internal/session/reconnect.go#L102-L106), [reconnect.go:256-258](server/internal/session/reconnect.go#L256-L258), [reconnect.go:315](server/internal/session/reconnect.go#L315).

2. **AC2 ‚Äî auto-start broadcast/status flip gated on `StartGame` success** _(resolves: "Auto-start broadcasts `system:game_started` even when `gameStarter.StartGame` failed", spec-quickplay-auto-seat-on-join review 2026-04-29)_

   **Given** an auto-start path attempts `gameStarter.StartGame` ‚Äî both [room/handler.go:670-731](server/internal/room/handler.go#L670-L731) (`SelectSeat`, 4th-seat) and [room/handler.go:1287-1298](server/internal/room/handler.go#L1287-L1298) (`QuickPlay`, last-seat),
   **When** `StartGame` returns a non-nil error,
   **Then** `system:game_started` is **NOT** broadcast,
   **And** the room's status is **NOT** flipped to `"playing"` (i.e. the tx that set status must be rolled back, OR the status update is deferred until after `StartGame` succeeds outside the tx ‚Äî pick one approach and apply it to both sites),
   **And** the four would-be participants receive a clear error toast (deliver via existing `ws.ErrorInvalidAction` or a new `ws.ErrorGameStartFailed` event ‚Äî see Dev Notes for choice rationale) instead of being navigated to a non-existent `/game/{id}`,
   **And** `playerCount`, seat assignments, and other tx-staged data remain consistent (no partial writes survive a failed start).

3. **AC3 ‚Äî `LeaveRoom` aborts when auto-start is in progress** _(resolves: "`LeaveRoom` does not gate on `room.Status == "waiting"` under lock", spec-quickplay-auto-seat-on-join review 2026-04-29)_

   **Given** a player calls `LeaveRoom` ([room/handler.go:443-542](server/internal/room/handler.go#L443-L542)) while another player is concurrently in the auto-start tx for the same room,
   **When** the leave's `RunInTransaction` body executes,
   **Then** the leave path MUST re-fetch the room **inside** the tx and check `room.Status == "waiting"`,
   **And** if the status has flipped to `"playing"` (i.e. a concurrent auto-start has already begun), the leave returns `apperr.ErrGameAlreadyStarted` (new sentinel ‚Äî see Dev Notes) and aborts WITHOUT removing the player or decrementing the counter,
   **And** `gameStarter.StartGame` never observes a `seatInfo` array containing a player who has already left,
   **And** the same gate is applied to `SelectSeat`'s leave-equivalent paths if any exist (audit `SelectSeat` for symmetry ‚Äî see Dev Notes).

4. **AC4 ‚Äî match record persisted before `event:match_end` broadcast** _(resolves D101)_

   **Given** [server/internal/session/manager.go:728-776](server/internal/session/manager.go#L728-L776) `handleMatchEnd` runs at natural match-end OR via accepted surrender,
   **When** the match record (and any settlement-relevant rows the broadcast implies exist) is persisted,
   **Then** persistence (`m.matchRepo.CreateWithHands(...)` AND `m.roomUpdater.UpdateRoomStatus(...)`) MUST complete BEFORE `BroadcastToUsers` for `event:match_end`,
   **And** a client that receives `match_end` and immediately calls a hypothetical `GET /api/v1/matches/:id` endpoint MUST find the row,
   **And** `event:match_end` is still emitted exactly once per match (don't accidentally drop it if the persist fails ‚Äî see Dev Notes for the failure-mode contract: persist failure is logged but the broadcast still fires so clients aren't stuck on the table forever).

   **Note:** the broadcast call lives at the trailing end of `broadcastActionResult` for the natural-end path and inside the surrender path's match-end branch. The fix is to move persistence ABOVE the broadcast in `handleMatchEnd` and ensure all invocation paths route through `handleMatchEnd` rather than emitting `match_end` directly.

5. **AC5 ‚Äî QuickPlay retries on `ErrRoomFull`, not only on code/name collision** _(resolves D29 + pickFirstEmptySeat fallback gap, spec-quickplay-auto-seat-on-join review 2026-04-29)_

   **Given** the QuickPlay retry loop at [room/handler.go:1113-1191](server/internal/room/handler.go#L1113-L1191),
   **When** the inner tx returns `apperr.ErrRoomFull` (raised by [pickFirstEmptySeat](server/internal/room/handler.go#L1328-L1341) when `player_count` denormalized counter says the room has free seats but every seat row is occupied ‚Äî i.e. counter drift),
   **Then** the loop MUST retry on `ErrRoomFull` in addition to `ErrRoomCodeTaken` / `ErrRoomNameTaken`,
   **And** counter drift surfaces as a successful join into a different/new room ‚Äî never as an opaque 5xx to the user,
   **And** `maxRetries` is honoured (counter drift in many rooms must still terminate with a clean error after the retry budget is exhausted; existing `maxRetries` guard does this ‚Äî verify behaviour preserved).

6. **AC6 ‚Äî logout clears `gameStore` so reconnect redirect cannot fire stale** _(resolves D66)_

   **Given** the 401 interceptor in [client/src/shared/api/axiosClient.ts:144-164](client/src/shared/api/axiosClient.ts#L144-L164) calls `useAuthStore.getState().logout()` after a refresh failure,
   **When** `authStore.logout()` runs ([client/src/shared/stores/authStore.ts:23-28](client/src/shared/stores/authStore.ts#L23-L28)),
   **Then** `useGameStore.getState().clearGame()` MUST also be invoked (and `useRoomLobbyStore`/`useChatStore` reset if they hold session-scoped state ‚Äî audit and apply consistently; see Dev Notes),
   **And** subsequent `useReconnectionRedirect` reads ([client/src/shared/hooks/useReconnectionRedirect.ts](client/src/shared/hooks/useReconnectionRedirect.ts)) find a clean store and DO NOT redirect a freshly re-logged-in user to a finished game's `/game/{id}` page,
   **And** the same store-clear executes on **manual logout** (the existing user-initiated path), so behaviour stays uniform across the two logout entry points.

7. **AC7 ‚Äî async session-manager tests no longer rely on `time.Sleep`** _(resolves D100)_

   **Given** the new race-path tests added by AC1‚ÄìAC4 plus the existing `time.Sleep`-based tests in [server/internal/session/manager_test.go](server/internal/session/manager_test.go) (~20 sites; see [grep above](#dev-notes)),
   **When** these tests run on loaded CI (Windows runners particularly),
   **Then** they MUST use a deterministic synchronisation primitive ‚Äî channels, `sync.WaitGroup`, or a small `waitFor(condition, timeout)` poll helper ‚Äî instead of fixed-duration sleeps for **all NEW tests added by this story**,
   **And** at minimum, replace `time.Sleep` in test sites that this story already touches (don't open the scope to a project-wide `time.Sleep` rewrite ‚Äî that's a follow-up sweep). The "touched" rule: any `time.Sleep` in a test function whose body this story modifies is in scope; others stay. Document the helper in a header comment so future tests adopt the same primitive.

## Dev Notes

### Big Picture ‚Äî This Is a Hardening Story, Not a Feature Story

Epic 8.5 exists for one reason: **Epic 9 introduces atomic match-end transitions (coin settlement, XP grant, honor delta) that share code paths with the bugs this story fixes.** Building wallet/XP/honor on top of a `handleMatchEnd` that broadcasts before persisting (D101), or a `LeaveRoom` that races auto-start (Task 3), would compound the same risk class ‚Äî except now with money in the mix.

### Architecture Patterns to Follow

- **Pure-function rules engine, untouched.** None of these fixes touch `internal/game/`. The story is about the *orchestration layer* (session manager, room handler, transaction boundaries, broadcast plumbing).
- **Atomic transactions own the truth.** AC2 and AC3 both hinge on placing the load-bearing read/write INSIDE `RunInTransaction`. The pre-tx `room` capture in `LeaveRoom` is fine for read-only display data (username, owner ID for the broadcast) but cannot be the source of truth for status ‚Äî that's why AC3 mandates a re-fetch inside the tx.
- **Persist before broadcast.** Architecture rule: WS broadcasts are observable side effects; persist authoritative state before clients can react. This is the AC4 invariant in one line.
- **Server is the authority.** The auth-store/game-store fix in AC6 is a UX-correctness fix ‚Äî the server already correctly rejects actions on ended games, but the client should not navigate the user to a game page that no longer exists for them.
- **One sentinel per error class.** `ErrGameAlreadyStarted` is its own sentinel, not a string-matched variant of `ErrInvalidAction`. This pays off in client-side toast specificity (Task 3.6) and in future story handlers.
- **Multi-event broadcasts as separate ordered messages.** AC2 emits a single `error:game_start_failed` to the four would-be participants; do NOT bundle it with any other event. Same rule as Story 8.3 (emote) and 8.2 (surrender).
- **Defence-in-depth client guards.** AC6 makes `gameState === null` the post-logout invariant; the existing `useReconnectionRedirect` early-return on null `gameState` is the second line of defence (already in place ‚Äî no change needed).
- **`data-testid` discipline** continues to apply ‚Äî any new test selectors use stable testids, not Tailwind classes (project rule).

### Backend Flow ‚Äî Key Race-Critical Paths

#### AC1 ‚Äî Cancel-then-rearm timer choreography (current state)

```text
HandleAction (manager.go:131)
  ‚îî‚îÄ session.mu.Lock()
  ‚îî‚îÄ session.cancelTurnTimer()        ‚Üê timer.go:5 (Stop() only, no bump)
  ‚îî‚îÄ ApplyAction ‚Üí newState
  ‚îî‚îÄ if normal-turn:
       m.startTimerLocked(session)    ‚Üê bumps timerGeneration, starts AfterFunc
  ‚îî‚îÄ session.mu.Unlock()

handleTimerExpiry (called by AfterFunc, in goroutine)
  ‚îî‚îÄ session.mu.Lock()                ‚Üê may block here past trigger
  ‚îî‚îÄ if session.timerGeneration != generation: return ‚Üê THE GUARD
  ‚îî‚îÄ ... auto-play ...
```

The bug: between `cancelTurnTimer()` (which calls `Stop()` but doesn't bump `timerGeneration`) and the next `startTimerLocked` (which DOES bump), there is a window where a stale goroutine that already raced past `Stop()` and is blocked on `mu.Lock()` will, when it acquires the lock, find `timerGeneration` unchanged and pass the staleness check. Today this is harmless because `handleTimerExpiry`'s switch falls through to `default: return` for the post-cancel state ‚Äî but a future patch that adds work above the switch would fire spuriously.

**The fix** moves the bump into `cancelTurnTimer()` itself, making "cancel == invalidate stale callbacks" a property of the function rather than the caller's responsibility.

#### AC4 ‚Äî Current `match_end` broadcast/persist ordering

```text
HandleAction (manager.go:131)
  ‚îî‚îÄ ApplyAction ‚Üí newState (Phase = match_end)
  ‚îî‚îÄ session.gameState = newState
  ‚îî‚îÄ session.mu.Unlock()
  ‚îî‚îÄ broadcastActionResult(...)        ‚Üê emits event:match_end at L505-508 (natural) or L622-630 (surrender)
  ‚îî‚îÄ if newState.Phase == PhaseMatchEnd:
       handleMatchEnd(...)             ‚Üê persists, AFTER broadcast already fired
```

The two broadcast sites are concrete: [manager.go:505-508](server/internal/session/manager.go#L505-L508) for the natural-end path (case `ActionPlayCard`) and [manager.go:622-630](server/internal/session/manager.go#L622-L630) for the surrender-accept path. `handleMatchEnd` lives at [manager.go:728-776](server/internal/session/manager.go#L728-L776).

**The fix** moves both `event:match_end` broadcasts OUT of `broadcastActionResult` and INTO `handleMatchEnd` AFTER `m.matchRepo.CreateWithHands(...)`. `broadcastActionResult` still owns `event:card_played`, `event:trick_resolved`, `event:hand_scored` ‚Äî it just stops owning `event:match_end`. The note at [manager.go:623-625](server/internal/session/manager.go#L623-L625) ("Emit event:match_end FIRST...handleMatchEnd...runs after this for persistence") becomes stale and must be removed/rewritten.
